# UZI-Skill 文章设计文档

## 背景

本次任务要围绕 `UZI-Skill` 产出一篇完整文章，并同时生成两个发布版本：

1. Hexo 博客使用的 Markdown 成稿
2. 可直接复制到微信公众号编辑器的 HTML 成稿

文章目标不是写成“AI 带你赚钱”的荐股内容，而是写成一篇带有强传播钩子、但依然符合微信平台合规边界的工具体验分享。用户已经确认本篇站位为“我亲自试过”的真实体验视角，但全篇会严格避免收益承诺、买卖建议和诱导性表达。

项目内已经存在一批未跟踪的 `UZI` 相关文章图片，且本地有 `../UZI-Skill` 仓库可直接读取，后续制作会优先复用这些真实素材，再补充必要的截图和 AI 生成封面。

## 目标

交付一套完整、可发布、风格统一的内容资产：

1. 一篇适合当前博客风格的中文文章
2. 一份适合微信公众号粘贴发布的 HTML
3. 一组用于博客和公众号共用的配图资产
4. 一套明确的合规措辞与风险提示

最终成品要同时满足以下标准：

1. 具备明确标题钩子与较强点击欲
2. 叙事上有真实体验感，而不是纯资料搬运
3. 视觉上以真实截图为主，AI 图只承担封面和总结卡角色
4. 不落入“荐股”“收益暗示”“带单”表达
5. 能直接落到 Hexo 与公众号两个渠道，无需用户二次整理

## 目标读者

主目标读者已经确认：

- 对 AI 炒股感兴趣
- 但还没装过类似 skill
- 同时具备一定技术背景，至少不排斥 agent、CLI、插件、仓库安装这类概念

这意味着文章必须兼顾两件事：

1. 让非重度 agent 用户也能读懂“它是什么、值不值得试”
2. 不把内容写得像纯技术文档或纯项目 README

## 内容定位

文章最终定位为：

`高钩子工具体验分享 + 上手导读混合稿`

核心卖点不是“它准不准”，而是：

1. 它把哪些原本分散、费劲的股票研究流程收拢成了一个入口
2. 它为什么会让技术用户觉得有意思
3. 它的价值边界在哪里，为什么不能把它当成自动买卖指令机

文章不采用“媒体资讯整理”视角，也不采用“纯方法论科普”视角，而是以“我亲自试过一个在圈子里传播起来的 AI 炒股 skill”为入口展开。

## 叙事主线

主叙事确定为：

> 我最近试了一个很有讨论度的 AI 炒股 skill。它最让我在意的，不是神奇收益故事，而是它把原本碎片化的研究流程组织成了一套可直接运行的分析系统。它更像一个股票研究助手，而不是一个替你做决定的自动投顾。

全篇围绕这条主线推进，避免叙事偏向以下两种错误方向：

1. 把文章写成“项目 README 中文复述”
2. 把文章写成“夸张神化式 AI 炒股种草”

## 文章结构

文章结构已经确认采用六段式：

1. 开头钩子
   用“我最近试了一个 AI 炒股 skill”切入，快速抛出项目的完成度与有趣点。
2. 先说结论
   明确指出它更像研究助手，不适合被理解为自动买卖建议机。
3. 它到底能做什么
   介绍多维分析、报告产出、命令体系、适配多种 agent 等能力。
4. 我为什么觉得它值得推荐
   从研究流程压缩、信息整理、多视角输出、上手体验等角度说明价值。
5. 谁适合试，谁不适合试
   主动筛选受众，增强可信度，同时降低泛化承诺风险。
6. 风险与边界
   收口到合规提醒和使用边界，避免形成荐股语境。

这个结构会延续当前博客已有文章的写法习惯，即：

- 先给判断
- 再分层拆解
- 最后明确适用人群与边界

## 标题策略

用户在视觉风格上选择了更强钩子的路线，但要求继续满足微信合规。因此标题策略确定为：

`高点击、低违规`

标题需要包含三类信息中的至少两类：

1. `AI 炒股 Skill`
2. `我亲自试了 / 最近试了`
3. `研究流程 / 研究助手 / 分析系统`

标题禁止出现以下方向：

1. 收益承诺：稳赚、翻倍、躺赚、暴富、闭眼买
2. 明确买卖诱导：抄底、跟买、带单、提示买点
3. 过度神化：封神、神器带飞、自动赚钱机器

实际执行时会给出 4 到 5 个候选标题，再从中选出一版用于博客和公众号共用；如需要，也可以保留一个更稳的备用标题。

## 视觉与封面策略

视觉方向确定为：

`强钩子封面 + 克制信息密度`

封面不做“暴涨线”“收益数字”“财富神话”表达，而是做成“AI 研究战情室 / 分析工作台”风格，元素包括：

1. AI 分析界面或报告卡片
2. 终端或命令行元素
3. 图表、评分、股票代码等研究符号
4. 偏深色或高对比度的专业氛围

视觉上强调“研究系统感”，不强调“赚钱结果感”。

## 配图策略

正文配图计划固定为 4 张，避免公众号过长、过花：

1. 封面图
   用于头图和博客封面，可由 AI 生成，但要贴近工具/研究氛围。
2. 能力总览图
   把项目核心能力压缩成一张信息卡，适合公众号中段承接。
3. 实测截图
   优先放真实项目截图，如报告页、关键命令输出、分析界面。
4. 适合谁 / 不适合谁总结卡
   用一张高信息密度的总结图强化分享价值和边界感。

配图优先级明确如下：

1. 优先复用 `../UZI-Skill` 仓库现成公开截图与图片
2. 复用论坛原帖中可公开引用的素材
3. 复用当前项目 `source/img/posts` 中已存在的 `uzi-*` 图片
4. 对缺口部分再用 `baoyu` 图片技能补齐

AI 生成图只承担：

1. 封面图
2. 信息总结卡

不会用 AI 图伪造产品界面或代替真实报告截图。

## 渠道适配

### 博客版本

博客版本使用 Markdown，落在：

- `source/_posts/<slug>.md`

图片放在：

- `source/img/posts/`

文风与现有博客文章保持一致：

1. 正文直接、判断明确
2. 不使用过度营销腔
3. 适当保留主观体验
4. 标注必要参考链接

### 微信版本

微信公众号版本使用单独 HTML 文件，要求：

1. 可直接复制到公众号编辑器
2. 样式适配微信正文宽度和字体环境
3. 段落更紧凑，避免博客式长段压迫感
4. 对外链进行适度收束，必要时做文末引用而非正文硬插

## 合规策略

本次内容以微信平台合规为硬边界。执行时必须遵守：

1. 不承诺收益
2. 不提供具体买卖建议
3. 不鼓动开户、跟单、群带单
4. 不使用“暴富”“躺赚”“抄底神器”等高风险词
5. 不把 AI 输出包装成确定性结论

全篇会固定设置边界提示，建议放在文章中后段和结尾至少各出现一次。基准表述如下：

> 本文仅为工具体验与研究流程分享，不构成任何投资建议，相关内容仅供参考。

如果正文中涉及个股、报告样例或分析结果，也会补充“仅为体验展示，不构成推荐”的局部说明。

## 素材来源与事实基线

正式执行时将以以下来源为事实基线：

1. `../UZI-Skill` 本地仓库内容
2. `UZI-Skill` GitHub README 与仓库内公开文档
3. `Linux.do` 原帖公开内容
4. 当前博客仓库内已存在的 `uzi-*` 相关图片

如果某个数字、命令、版本信息在多个来源之间不一致，以执行当时最新仓库内容为准，并在写作时尽量避免使用高时效、易过期的具体数字，除非已复核。

## 执行产物

正式执行阶段要产出以下内容：

1. 一篇 Hexo Markdown 文章
2. 一个公众号 HTML 文件
3. 一张封面图
4. 三张正文配图或整理图
5. 所有最终使用图片的本地落盘版本

如果现成截图足够，AI 生成图片数量可以减少，但封面图原则上仍建议单独做，以保证整体包装完整度。

## 执行步骤

正式执行时按以下顺序推进：

1. 读取并整理仓库与论坛原始资料
2. 确认实际可用截图与图片资产
3. 写出博客版 Markdown 初稿
4. 在初稿中标定配图插入位
5. 整理真实截图，补做缺失图片
6. 生成封面图和总结卡图
7. 将 Markdown 转为公众号 HTML
8. 做微信合规收口与最终通读

这个顺序能确保内容先定，再做视觉，不会出现图片先行导致正文被动适配的问题。

## 风险与应对

当前主要风险有四类：

1. 仓库内容更新较快
   应对：写作前重新核对本地仓库 README 和关键命令。
2. 论坛内容抓取不稳定
   应对：以 GitHub 仓库为主事实源，论坛内容只做观点补充。
3. 微信对“炒股”类内容较敏感
   应对：全篇固定压低收益暗示，突出“研究流程”“工具体验”“仅供参考”。
4. 现有图片未跟踪、来源需确认
   应对：正式使用前逐张核对来源，优先使用仓库公开图片和本地仓库截图。

## 不做的事

为保证范围清晰，本次不做以下内容：

1. 不做投资结论正确性评测
2. 不验证项目长期收益能力
3. 不写成安装文档全文翻译
4. 不扩展成多项目横评
5. 不在本阶段处理发布到公众号后台的自动化流程

## 验收标准

完成后，以以下标准判断是否达标：

1. 用户拿到后可直接在博客发布 Markdown
2. 用户拿到后可直接复制 HTML 到公众号编辑器
3. 图片与正文风格统一，不依赖远程图床临时链接
4. 标题与导语有传播力，但没有明显违规措辞
5. 文中已明确说明“仅为工具体验与研究流程分享，不构成投资建议”
